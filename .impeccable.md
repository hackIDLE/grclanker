# grclanker Design Context

## Users

**Primary:** GRC engineers, compliance automation builders, FedRAMP/CMMC assessors who write code or use AI agents. They know what a spec is. They want to grab the file and go. Respect their time — don't explain things they already understand.

**Secondary:** Security teams evaluating whether spec-driven compliance is viable for their org. They're browsing, trying to understand the format, maybe feeding one spec to an agent to see what happens.

**Mindset:** Both task-focused ("give me the CrowdStrike spec") and exploratory ("what is this, and can I use it?"). The site must serve both without slowing down either.

**Context:** These users live in terminals, IDEs, and agent interfaces. They're skeptical of marketing sites. They respect tools that work well and don't waste their time.

## Brand Personality

**Technical. Direct. No-BS.**

Like a well-run cockpit — functional, no wasted space, everything has a purpose. The site should feel like an instrument panel: dense with information, immediately operable, zero decorative fluff that doesn't serve a function.

The cockpit metaphor is literal: the visual theme uses a retro sci-fi spacecraft cockpit aesthetic with HUD elements, instrument panels, and status indicators. This isn't decoration — it's the brand language. Every visual element should feel like it belongs in a cockpit.

## Aesthetic Direction

**Theme:** Catppuccin Frappé (dark only). Pastel-on-dark with warm, readable colors.

**Typography:** Google Sans Mono exclusively. Monospace everything — headings, body, code, labels. The uniformity reinforces the technical, cockpit-instrument feel.

**Visual language:** HUD elements (corner brackets, scan lines, status LEDs, gradient dividers, uppercase tracking labels). Panels and viewscreens as surface metaphors. Dark steel backgrounds with soft blue (`#8caaee`) as the primary accent.

**References that capture the feel:**
- Retro anime cockpits (the hero image)
- Flight instrument panels — dense, functional, high-information-density
- Terminal UIs like lazygit, btop — keyboard-driven, no mouse needed

**Anti-references (explicitly NOT this):**
- SaaS landing pages: no hero CTAs, no "trusted by" logos, no gradient buttons, no pricing tables, no testimonial carousels
- Generic developer docs: not ReadTheDocs, not Docusaurus default, not a wiki. Has its own identity.
- AI-slop aesthetics: no Inter font, no purple gradients on white, no cookie-cutter component libraries

## Design Principles

### 1. The spec is the product
The raw URL is the most important element on every page. Everything else exists to help users find, copy, and use that URL. Never bury the spec behind clicks, signups, or marketing copy.

### 2. Keyboard-first interaction
The site should be operable without a mouse. Global search via `/` or `Cmd+K`. Arrow keys to navigate the spec catalog. `Enter` to open a spec. `c` to copy the raw URL. The cockpit metaphor demands keyboard fluency.

### 3. Information density over whitespace
These users read dashboards and terminals all day. Don't spread thin content across large spaces. Pack information tight, use clear hierarchy (HUD labels, font weight, color) to make it scannable, and let the content breathe through consistent spacing — not empty space.

### 4. Every visual element earns its place
HUD corners, scan lines, status LEDs, gradient dividers — each serves a function (grouping, status indication, section separation, depth). If a visual element doesn't help the user find or use a spec, remove it.

### 5. Agent-agnostic by default
Never assume which coding agent a user has. Show examples for Claude Code, Codex, Gemini CLI, Cursor, Copilot, Aider, and generic patterns. The raw URL works with everything — make it the hero.

## Technical Constraints

- **Framework:** Astro 6.x (static site generation, zero client JS by default)
- **Styling:** Tailwind CSS 3.x with Catppuccin Frappé tokens
- **Font:** Google Sans Mono (self-hosted, 400/500/700 weights)
- **Images:** WebP for generated assets, JPG for OG card
- **Content:** 31 spec.md files with YAML frontmatter, parsed at build time
- **Performance:** 34 pages build in ~800ms. Keep it fast.
- **Accessibility:** WCAG AA contrast ratios, keyboard navigable, respect `prefers-reduced-motion`
